About the Reaction Time project

The Reaction Time project is the first project to be added to my portfolio. For this project I used HTML, CSS and TypeScript.

The goal

The goal was to recreate the Reaction Time test by Human Benchmark with some of my own ideas. Since I don't have access to a database through Github Pages, this would be the perfect opportunity to learn about using localstorage.

Lessons, Challenges & Hurdles

  • Lesson: Use localstorage for persistent data for returning visitors
  • Lesson: Design a proper way of showing data
  • Challenge: I started writing out the code to get the basic logic working. I got it working fast but it, of course, turned into a bit of spaghetti code. So to clean up and learn about software architecture, I researched and refactored the code according to a the MVC model. But because the project is relatively small, making the code pure MVC would be overkill and lack code clarity. Finding the right balance of using the MVC model while also keeping the code easy-to-read was challenging.
  • Hurdle: The reaction times initially were ~100ms slower than on other websites. There were 3 issues that needed to be solved:
    1. I registered a click event, which means that the click is only registered on releasing the mouse button. Changing the addEventListener('click', ...) into addEventListener('pointerdown', ...) reduced the biggest gap. Pointerdown is different from mousedown, which could also be an option, but this would not always work on mobile. Pointerdown works for both desktop and mobile.
    2. I used Date.now() for setting the start time, when banner turned green, and the end time, when the user clicked. For interactive games where timing matters, you want to use performance.now() as it is more precise due to representing time as float-point number, and is not affected by system adjustments due to being a monotonic clock
    3. The last issue I had was that I set the start time before I rendered the banner green. This resulted in the 'timer' basically running before the user could even see the changes. Simply moving the render before setting the start time was only partly a fix, because of how browser event loops work. There is a gap between when running the code and when the pixels change color. So to make sure that the time starts as soon as the pixels change color, we use requestAnimationFrame.
